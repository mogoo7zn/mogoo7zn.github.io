## 项目概览

这次比赛围绕 OpenHarmony 设备上的离线多模态大模型应用展开。我们的队伍“炸薯丸”以 DAYU200 开发板为主要实验平台，目标是在资源受限的端侧设备上完成大模型推理能力落地，并进一步探索多模态与系统级能力的整合

我在项目中主要参与端侧部署方案调研、模型推理链路实现与决赛阶段的多模态探索，同时承担队长职责，推进任务拆分、阶段统筹与最终答辩材料整理

## 初赛：完成端侧 LLM 基础链路

初赛阶段的核心任务，是在 OpenHarmony 环境中打通从应用界面到模型推理的完整链路。我们先把问题收敛到“让 LLM 稳定在板端跑起来”，再逐步考虑多模态和性能优化

### 初赛目标

- 在 DAYU200 上完成 OpenHarmony 环境搭建与应用开发调试
- 设计一套基于 ArkUI 与 ArkTS 的聊天交互界面
- 将第三方推理库移植到设备端，实现 LLM 本地推理
- 打通应用层与推理层之间的数据交互与结果展示

### 初赛技术选型

我们对 `MindSpore Lite`、`llama.cpp` 和 `MNN` 三条路线做了较系统的调研

- `MindSpore Lite` 与 OpenHarmony 生态结合紧，但对 LLM 的支持不够完整
- `llama.cpp` 对大语言模型支持成熟，量化与推理链路清晰，因此成为初赛阶段的主方案
- `MNN` 提供了轻量推理框架与一定的端侧潜力，但在 OpenHarmony 上的部署和多模态性能仍有限制

### 初赛实现方式

为了让推理能力真正嵌入应用，我们采用了分层设计：

- 前端界面使用 ArkUI + ArkTS，负责聊天交互与状态展示
- 中间层使用 NAPI 连接 ArkTS 与原生 C++，承担接口桥接工作
- 推理后端采用 `llama.cpp`，负责模型加载、推理执行与流式输出

在工程实现上，我们通过 OpenHarmony NDK 将 `llama.cpp` 及相关动态库编译并接入应用，再利用异步线程与回调机制，将推理结果持续返回给界面层，实现接近实时的对话体验

## 决赛：从单模态推理走向多模态探索

进入决赛后，项目目标从“能跑”转向“更强、更完整”。在保留已有 LLM 端侧能力的基础上，我们开始尝试多模态大模型与操作系统能力协同，并针对真机环境继续推进性能优化

### 决赛重点

- 围绕华为真机环境继续完善端侧推理方案
- 探索视觉编码与语言模型结合的多模态输入处理
- 尝试利用设备 GPU 算力提升推理速度
- 面向系统级能力封装更通用的大模型调用接口

### 关键挑战

- 端侧多模态模型对显存和算力要求远高于纯文本模型
- 设备资源受限，必须严格控制模型大小、上下文长度与中间缓存
- OpenHarmony 生态下的第三方推理框架适配工作量较大
- 需要在演示稳定性、速度和功能完整度之间做实际取舍

### 阶段成果

- 完成了板端与真机环境下的大模型部署与应用整合
- 保持了从界面输入到推理输出的完整闭环
- 对多模态方案进行了原型级探索与适配尝试
- 在决赛阶段实现了约 `15 token/s` 的端侧推理能力

## 队长视角

这次比赛最有价值的不只是“把模型部署上去”，而是如何在有限时间里，把调研、实现、验证、展示四件事压缩进同一条推进链路。作为队长，我需要持续判断哪些方向值得投入，哪些目标必须阶段性收缩，并确保队员之间的接口边界足够清晰

相比单纯写代码，真正困难的部分往往是把方案讲明白、把问题定位准、把任务节奏稳住。比赛过程让我更明确地认识到，工程落地能力本质上是技术判断、沟通表达和项目管理能力的结合

## 收获

- 更系统地理解了端侧 LLM 部署中的模型选型、量化、推理框架与资源约束问题
- 积累了 OpenHarmony 场景下 ArkTS、NAPI、C++ 协同开发的实践经验
- 对多模态大模型在嵌入式设备上的落地难点有了更具体的认识
- 在高压竞赛环境中完成了团队协作、任务拆分与技术答辩的完整闭环
